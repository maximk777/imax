Архитектурные подходы

Как организовать слои приложения для Desktop + Web + Mobile

A

Monolith Core — рекомендую

Единое ядро (core lib) содержит всю бизнес-логику, P2P, крипто, хранилище. Dioxus UI — тонкая оболочка, которая вызывает core через Rust API.

┌─────────────────────────────────┐
│ Dioxus UI (Desktop/Web/Mobile) │
└──────────────┬──────────────────┘
│ Rust API calls
┌──────────────▼──────────────────┐
│ dchat-core (library crate) │
│ ┌────────┐ ┌──────┐ ┌───────┐ │
│ │Network │ │Crypto│ │Storage│ │
│ │ (iroh) │ │(E2E) │ │(SQLite)│ │
│ └────────┘ └──────┘ └───────┘ │
└─────────────────────────────────┘

Pros

  • Один codebase — UI просто рендерит состояние
  • Проще тестировать ядро без UI
  • Все платформы получают одинаковую логику
  • Быстрый путь к MVP

Cons

  • Веб-платформа потребует relay (нет прямых сокетов)
  • Нужна условная компиляция для платформ
B

Core + Platform Adapters

Ядро + платформенные адаптеры. Core определяет трейты (NetworkAdapter, StorageAdapter), каждая платформа реализует их по-своему. Веб — через WebRTC/relay, десктоп — через iroh напрямую.

┌────────────────────────────────────────┐
│ Dioxus UI │
└──────────────┬─────────────────────────┘
┌──────────────▼─────────────────────────┐
│ dchat-core (traits + logic) │
└──┬──────────────┬──────────────┬───────┘
│ │ │
┌──▼───┐ ┌────▼────┐ ┌────▼────┐
│Desktop│ │ Web │ │ Mobile │
│ iroh │ │WebRTC/ │ │ iroh │
│direct │ │ relay │ │ direct │
└───────┘ └─────────┘ └─────────┘

Pros

  • Чистое разделение платформенного кода
  • Каждая платформа использует оптимальный транспорт
  • Легче масштабировать на новые платформы

Cons

  • Больше абстракций и кода на старте
  • Over-engineering для MVP
  • Тестировать нужно каждый адаптер отдельно
